# ggrigggri_compose 프로젝트 구조 분석 및 개선 가이드

## 1) 현재 구조 요약
- 모듈 구성: `:app`, `:feature:ggriggri_ui`, `:feature:common_ui`, `:data`, `:domain`
- 현재는 `app / feature / data / domain` 분리가 되어 있어 큰 방향은 적절합니다.
- Compose + Hilt + Navigation 조합도 실무에서 많이 쓰는 안정적인 구성입니다.

## 2) 우선 개선 포인트 (중요도 순)
1. **모듈 경계 위반(가장 중요)**
- `feature` 모듈이 `data` 구현에 직접 의존하고 있습니다.
- 실무 권장: `feature -> domain`, `data -> domain` 방향만 허용.

2. **ViewModel 내부 `runBlocking` 사용**
- UI 멈춤/프레임 드랍 가능성이 있어 권장되지 않습니다.
- `viewModelScope + suspend/Flow` 기반으로 치환 필요.

3. **중첩 `collect` 과다**
- `collect` 안에 또 `collect`가 겹치면 로직 추적이 어려워지고 취소 제어가 힘듭니다.
- `flatMapLatest`, `combine`, `first`, `stateIn`으로 단순화 필요.

4. **패키지/경로 네이밍 불일치**
- 실제 파일 경로와 package 선언 규칙이 일부 달라 유지보수 비용이 증가합니다.
- 경로와 package를 1:1로 맞추는 것이 좋습니다.

5. **미사용/고립 모듈 흔적**
- include되지 않은 모듈/폴더(`app/common`, `core`)가 존재합니다.
- 유지할지 제거할지 의사결정 필요.

6. **빌드 설정 불일치**
- 모듈별 `compileSdk`가 35/36 혼재.
- 버전 기준 통일이 필요합니다.

7. **MainActivity 권한 처리 코드 불완전**
- 권한 런처 초기화 코드가 주석인데 호출 함수는 살아 있어 런타임 리스크가 있습니다.

8. **테스트 부족**
- 대부분 Example 테스트 템플릿 수준.
- 핵심 로직 테스트가 필요합니다.

## 3) MVC + MVVM 공존 전략 (Compose 기준)
### 핵심 원칙
- **비즈니스 로직은 MVVM**
- **플랫폼 제어(권한/SDK 콜백/Activity Result)는 MVC 성격의 Controller로 분리**

### 역할 분리
- `View(Composable)`: 상태 표시 + 사용자 이벤트 전달
- `ViewModel`: `UiState`, `UiEvent`, `UiEffect`, 비즈니스 흐름
- `Controller`: 안드로이드 플랫폼 상호작용(권한 요청, 콜백 브리지)
- `Model`: domain model / usecase

이렇게 두면 MVC와 MVVM을 동시에 쓰더라도 책임 충돌이 줄어듭니다.

## 4) 단계별 개선 로드맵
### 1단계: 의존성 경계 정리
- 목표: `feature`가 `data`를 몰라도 동작하도록 변경
- 액션:
  - `StorageRepository` 인터페이스를 `domain`으로 이동
  - `feature`의 `com.ahn.data.*` import 제거
  - 구현체 바인딩은 `data` DI 모듈에서 담당

### 2단계: 패키지 구조 표준화
- 목표: 경로와 package 일치
- 액션:
  - 예: `screen/ui/main/ui/home` 경로면 package도 동일 계층
  - 기능 단위(`feature/home`, `feature/group`)로 수직 분할

### 3단계: ViewModel 패턴 통일
- 목표: 상태 관리 단순화
- 액션:
  - `UiState + onEvent()` 단일 진입점
  - `runBlocking` 제거
  - 중첩 `collect` 제거

### 4단계: Compose 화면 정리
- 목표: 재컴포지션/사이드이펙트 안정화
- 액션:
  - 화면 내부 과도한 로그 정리
  - 로딩 트리거는 `LaunchedEffect(Unit)` + ViewModel 이벤트 방식으로 통일

### 5단계: 테스트 도입
- 목표: 회귀 방지
- 액션:
  - ViewModel 상태 전이 테스트
  - Repository Fake 기반 UseCase 테스트
  - Navigation 스모크 테스트

## 5) Kotlin 예시
### ViewModel (MVVM 정석 형태)
```kotlin
data class HomeUiState(
    val isLoading: Boolean = false,
    val profiles: List<ProfileUi> = emptyList(),
    val error: String? = null
)

sealed interface HomeEvent {
    data object Load : HomeEvent
    data object Refresh : HomeEvent
}

@HiltViewModel
class HomeViewModel @Inject constructor(
    private val getHomeDataUseCase: GetHomeDataUseCase
) : ViewModel() {

    private val _uiState = MutableStateFlow(HomeUiState())
    val uiState: StateFlow<HomeUiState> = _uiState.asStateFlow()

    fun onEvent(event: HomeEvent) {
        when (event) {
            HomeEvent.Load, HomeEvent.Refresh -> load()
        }
    }

    private fun load() = viewModelScope.launch {
        _uiState.update { it.copy(isLoading = true, error = null) }
        runCatching { getHomeDataUseCase() }
            .onSuccess { data ->
                _uiState.update { it.copy(isLoading = false, profiles = data.profiles) }
            }
            .onFailure { e ->
                _uiState.update { it.copy(isLoading = false, error = e.message) }
            }
    }
}
```

### Composable (상태 소비)
```kotlin
@Composable
fun HomeScreen(
    viewModel: HomeViewModel = hiltViewModel(),
    onNavigateToRequest: () -> Unit
) {
    val state by viewModel.uiState.collectAsStateWithLifecycle()

    LaunchedEffect(Unit) { viewModel.onEvent(HomeEvent.Load) }

    when {
        state.isLoading -> CircularProgressIndicator()
        state.error != null -> Text(text = state.error!!)
        else -> HomeContent(
            profiles = state.profiles,
            onRequestClick = onNavigateToRequest
        )
    }
}
```
